# API Contract: Parser to Planner

This document defines the structured JSON output generated by the Parser (Language Module). The Planner (World Module) will consume this data to evaluate preconditions, execute actions, and update the world state.

## 1. Core Schema Definition

Every payload from the parser will follow this structure:

*   **`intent`** *(String | null)*: The action the user wants to perform.
    *   Allowed values: `PICKUP`, `PLACE`, `INSPECT`, `OPEN`, `CLOSE`
*   **`action_args`** *(Object | null)*: The physical parameters required to execute the intent.
    *   **`target`** *(String)*: The ID of the primary object to interact with.
    *   **`destination`** *(Object)*: Required **only** for the `PLACE` intent.
        *   **`relation`**: The spatial relationship (`INSIDE`, `NEXT TO`, `ON TOP OF`, `UNDER`).
        *   **`reference`**: The ID of the destination object.
*   **`status`** *(String)*: The resolution state of the parsed command.
    *   Allowed values: `RESOLVED`, `AMBIGUOUS`, `UNRECOGNIZED`
*   **`status_args`** *(Object | null)*: Context required **only** if the status is not resolved.
    *   **`target_candidates`** *(List of Strings)*: Possible object IDs if the target is ambiguous.
    *   **`reference_candidates`** *(List of Strings)*: Possible object IDs if the destination reference is ambiguous (only applies to `PLACE`).
    *   **`message`** *(String)*: The prompt to display to the user to resolve the error or ambiguity.

---

## 2. Execution Routing Rules

> **Important Note for the Planner:** The physical execution logic should **only** run if the `status` is `"RESOLVED"`. 

*   **If `status == "RESOLVED"`:** The Planner executes the action using `intent` and `action_args`.
*   **If `status == "AMBIGUOUS"` or `"UNRECOGNIZED"`:** The Planner will not be called. The Main Loop will intercept this payload and output the `message` to ask the user for clarification.

---

## 3. Example Payloads

### Example 1: Fully Resolved Command
*Generated when the parser perfectly understands the command and uniquely identifies the object.*
```json
{
  "intent": "PICKUP",
  "action_args": {
    "target": "obj_green_ball_2"
  },
  "status": "RESOLVED"
}
```

### Example 2: Ambiguous Destination
*Generated when the target is clear, but the destination object is ambiguous (e.g., "Put the red block in the box" when there are two boxes).*
```json
{
  "intent": "PLACE",
  "action_args": {
    "target": "obj_red_block_1",
    "destination": {
      "relation": "INSIDE",
      "reference": null
    }
  },
  "status": "AMBIGUOUS",
  "status_args": {
    "target_candidates": [],
    "reference_candidates": [
      "obj_box_1", 
      "obj_box_2"
    ],
    "message": "Which box do you mean? The wooden one or the metal one?"
  }
}
```

### Example 3: Unrecognized Command
*Generated when the parser fails to understand the grammar or vocabulary (e.g., "Dance on the table").*
```json
{
  "intent": null,
  "action_args": null,
  "status": "UNRECOGNIZED",
  "status_args": {
    "message": "I don't understand that command."
  }
}
```
